# Workflow Guide

The goal is not to build a vague second brain. The goal is to turn daily input into reusable business and technical assets that stay linked across accounts, contacts, projects, meetings, and issues.

## Four Core Streams

1. Technical knowledge lives in `03 Knowledge/Tech`
2. Industry knowledge lives in `03 Knowledge/Industry`
3. Account and relationship knowledge lives in `04 Clients`
4. Delivery and deployment history lives in `05 Projects`

## The Object Model

1. `Account`
   One company or customer entity such as `GreyOrange`
2. `Contact`
   One person under that account
3. `Project / Site`
   One opportunity, deployment site, or delivery object
4. `Meeting`
   One communication event
5. `Issue`
   One support, quality, product, or field problem

## Daily Workflow

1. Open `00 Dashboard/Start Here`
2. Open today's daily note and set three priorities
3. Write raw input into the daily note first
4. Use QuickAdd only when the record clearly deserves a formal note immediately
5. Before the day ends, move useful output into account, project, meeting, issue, or knowledge notes
6. Run Git backup

## Note-Taking Rules

- One note should primarily represent one object
- Write conclusions, not just links
- Accounts, contacts, projects, meetings, and issues should link to each other
- Put important fields in YAML when possible so Dataview can read them

## When To Create Which Note

- Technical designs, debugging lessons, reusable methods: `03 Knowledge/Tech`
- Market observations, competitors, policy, trends: `03 Knowledge/Industry`
- Companies, contacts, requirements, and momentum: `04 Clients`
- A concrete delivery target or deployment site: `05 Projects`
- A meaningful conversation, interview, or review: `06 Meetings`
- A support problem, field problem, or defect: `08 Issues`

## What Not To Do In Week One

- Do not create a giant tag taxonomy
- Do not create too many template types
- Do not migrate every old file at once
- Do not let AI replace structure; use AI after the structure is clear
